## Iteration 5: BullMQ Worker Integration (Completed)

### Summary
Implemented automatic document embedding processing via BullMQ Worker. Previously, documents were queued but never processed. Now they are processed automatically when uploaded.

### Changes Made

1. **Worker Manager Singleton** (src/lib/queue/worker-manager.ts)
   - Idempotent worker initialization via ensureWorkerStarted()
   - Graceful shutdown via stopWorker()
   - Singleton pattern prevents multiple workers

2. **Worker Startup Integration** (src/instrumentation.ts)
   - Worker starts on server initialization (highest priority fix)
   - Added stopWorker() to graceful shutdown sequence
   - Logs "embedding processor worker started" on app startup
   - Works in both dev and production environments

3. **API Route for Manual Control** (src/app/api/queue/start/route.ts)
   - POST /api/queue/start: Start worker on-demand
   - GET /api/queue/start: Check worker status
   - Useful for testing and admin operations

4. **Client-side Worker Initializer** (src/components/layout/WorkerInitializer.tsx)
   - Fallback initialization via useEffect on page load
   - Provides redundancy in case instrumentation hook fails

5. **Simplified Embedding Function** (src/features/documents/actions/embed-document.ts)
   - Removed internal retry loop (was causing double-retry pattern)
   - Removed 'use server' directive (function runs in Worker process)
   - Now relies solely on BullMQ retry logic (3 attempts, exponential backoff)
   - Single attempt per job, BullMQ handles retries
   - Cleaner architecture, predictable retry timing

### Verification

- ✅ Worker starts automatically on server startup
- ✅ Document status changes from 'waiting' to 'done' automatically (no manual API call needed)
- ✅ Chunks created and stored in ChromaDB (chunk_count: 1+)
- ✅ Semantic search returns relevant results (50%+ match)
- ✅ FTS search returns exact matches (100%)
- ✅ Hybrid search combines both
- ✅ TypeScript compilation: 0 errors
- ✅ E2E tests: All passing (71 tests)

### Architecture Notes

- Worker is started during Next.js instrumentation hook (nodejs runtime only)
- BullMQ handles all job retry logic with exponential backoff
- embedDocument() throws on failure, BullMQ catches and retries
- Graceful shutdown includes worker.close() for clean process termination
- Worker uses connection pooling from redis-config.ts

### Trade-offs Made

- Kept /api/documents/process-embeddings endpoint for admin/testing use (useful for batch re-processing)
- Worker startup via instrumentation means jobs must wait for app to start (but much more reliable than client-initiated)
- Client-side WorkerInitializer as fallback provides extra redundancy
